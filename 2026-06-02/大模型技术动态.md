# 大模型技术动态日报（2026-06-02）

## OpenAI frontier models 与 Codex 在 AWS 上 GA，企业可通过 Bedrock 与 GovCloud 路径接入

**入选理由**：OpenAI 官方于 2026-06-01 10:00 GMT 发布公告，确认 OpenAI frontier models 与 Codex 在 AWS 上一般可用。这属于头部模型厂商的云平台可用性变化，影响企业采购、治理和生产部署路径。

**技术细节**：公告给出两条接入路径：OpenAI models on Amazon Bedrock 用于在 AWS 原生安全与治理控制下构建 AI 应用；Codex on Amazon Bedrock 将软件工程 Agent 引入 AWS 环境，用于写代码、代码审查、调试与现代化改造。OpenAI 称该组合覆盖 AWS Commercial 与 GovCloud 区域。公告没有披露本次 AWS 渠道的新模型 ID、上下文长度、速率限制、价格或 benchmark 变化；企业落地时应以 Bedrock 控制台、AWS 价格页和具体模型卡为准。

**影响与处置**：对已在 AWS 上运行数据治理、身份、审计和采购流程的企业，主要变化是可以减少单独引入外部模型服务的合规摩擦。安全团队应重点核查 Bedrock 侧日志、IAM 权限、数据边界、模型调用审计、Codex 可访问代码仓库范围和代码审查输出的变更审批链路。公告还提到未来 Daybreak、cyber models 与 Codex Security 将通过 AWS 路径扩展，相关安全能力单独归入交叉动态。

**信息来源**：

- https://aws.amazon.com/bedrock/openai/
- https://developers.openai.com/codex/amazon-bedrock

## GitHub Copilot 改为全计划按 AI Credits 用量计费，并允许组织配置代码审查默认 Actions runner

**入选理由**：GitHub Changelog 于 2026-06-01 发布 Copilot 计费与计划更新，涉及所有 Copilot 计划的用量计费、代码审查运行资源和组织级预算控制，是开发者 AI 工具链的实质变更。

**技术细节**：自 2026-06-01 起，所有 Copilot 计划按 GitHub AI Credits 消耗计费，各计划包含月度用量。超出包含额度后，用户需要配置额外支出预算才能继续消费 AI Credits；个人计划的额外额度还可能受使用模式、账单历史和账号验证状态限制。Copilot code review 现在同时消耗 GitHub Actions minutes 与 AI Credits，默认使用标准 GitHub-hosted runner；组织管理员可设置组织级默认 runner，避免逐仓库配置。组织与企业计划新增用户级预算，可为所有用户设统一预算，也可覆盖特定用户集合。

**影响与处置**：工程平台团队需要把 Copilot code review 纳入 Actions runner 容量、费用和隔离策略，而不只是模型调用预算。建议检查组织级 runner 是否可能访问敏感网络或凭据，给 Copilot code review 单独设置预算和监控告警，并复核审查任务生成的日志保留策略。

- https://docs.github.com/copilot/concepts/billing/usage-based-billing-for-organizations-and-enterprises
- https://docs.github.com/copilot/how-tos/copilot-on-github/set-up-copilot/configure-runners

## GitHub Copilot individual non-enterprise 用户的 auto 模式可路由到 evaluation models

**入选理由**：GitHub Changelog 于 2026-06-01 公告，Copilot individual non-enterprise 用户可以访问 evaluation models，且这些模型可能被 Copilot auto model selection 选中。该变化影响模型选择透明度和企业外开发者的输出一致性评估。

**技术细节**：GitHub 表示 evaluation models 可能通过 Copilot auto model selection 提供给个人非企业用户。用户可在 GitHub Copilot settings 中关闭 auto 模式使用 evaluation models。公告没有披露 evaluation models 的具体模型 ID、训练来源、上下文长度、速率限制、价格或安全卡；因此只能将其视为模型路由策略变化，而不是确认的新模型发布。

**影响与处置**：对需要可复现代码生成结果或受合规约束的个人开发者，应关闭 evaluation models 或固定模型；对组织环境，应继续使用企业级模型规则限制可用模型集合，避免评估模型进入关键代码路径。
