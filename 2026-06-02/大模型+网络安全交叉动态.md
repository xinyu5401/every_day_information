# 大模型 + 网络安全交叉动态日报（2026-06-02）

## OpenAI on AWS 公告提及 Daybreak、cyber models 与 Codex Security 将走 AWS 企业治理路径

**入选理由**：OpenAI 2026-06-01 官方公告在 AWS GA 条目中专门列出“future availability for Daybreak”，并说明 Daybreak 包含 cyber models 与 Codex Security。这是头部模型厂商把网络安全专用模型/Agent 能力纳入云平台治理体系的实质信号。

**技术细节**：公告称 Daybreak 面向软件构建与防御场景，能力方向包括 secure code review、threat modeling、patch validation、dependency risk analysis、detection 与 remediation guidance，并计划把这些能力纳入日常开发循环。当前公告没有披露 Daybreak 的模型 ID、system/model card、上下文长度、工具调用权限、价格、速率限制、benchmark 或可用时间；因此不能视为已 GA 的安全模型发布，而应视为 AWS 渠道下的产品路线与可用性预告。

**代码 Agent 安全基线**：落地前应明确工具入口是 Bedrock、Codex on Bedrock 还是后续 Daybreak 服务；可访问资源包括代码仓库、依赖清单、漏洞扫描结果、构建日志和运行时告警；命令执行权限与凭据访问应默认隔离；审批机制应覆盖补丁生成、PR 创建、依赖升级和检测规则变更；日志字段至少记录模型/Agent ID、输入来源、工具调用参数、被访问仓库、生成补丁 diff、审批人、运行环境和回滚引用。

**影响与处置**：对安全团队，价值在于把威胁建模、依赖风险、补丁验证和检测建议接入同一开发工作流；风险在于代码 Agent 可能读取敏感源代码、凭据、漏洞详情和内部拓扑。建议先在低敏仓库试点，使用只读凭据、最小权限 IAM、隔离 runner、强制人工审批和可回滚 PR 流程；任何自动修复不得绕过代码所有者审查。

- https://aws.amazon.com/bedrock/openai/
- https://developers.openai.com/codex/amazon-bedrock

## Red Hat npm 供应链事件波及 MCP 相关包，MCP server 依赖链需要纳入凭据泄露排查

**入选理由**：Aikido 披露的 `@redhat-cloud-services` npm 污染事件中，受影响包列表包含 `hcc-feo-mcp`、`hcc-kessel-mcp`、`hcc-pf-mcp` 等 MCP 命名包。该事件本质是开发者供应链攻击，但会进入 Agent/MCP 工具生态的依赖面。

**技术细节**：攻击链使用 GitHub Actions OIDC trusted publishing 发布后门包，安装阶段通过 `preinstall` 执行载荷，窃取 CI/CD、云、包仓库、SSH、Kubernetes、Docker 与 `.env` 凭据。对 MCP 场景，风险不只在 npm install 主机本身：MCP server 往往被代码 Agent、IDE Agent 或内部自动化系统调用，可能持有工具凭据、内部 API token、仓库访问权限和长期上下文。如果受污染 MCP 包进入 server 镜像或本地开发环境，攻击者可能获得 Agent 工具链的上游凭据。

**代码 Agent/MCP 安全基线**：应记录 MCP server 包名与版本、安装来源、lockfile hash、镜像 digest、可访问工具列表、外部内容注入点、持久化存储位置、凭据挂载路径、Agent 审批策略和工具调用日志。对本次事件，重点查 `preinstall` 执行痕迹、异常 npm trusted publishing、异常 GitHub Actions workflow、`id-token: write` 权限、CI runner 文件系统中的云密钥和包仓库 token。

**影响与处置**：使用相关 MCP 包或 `@redhat-cloud-services` 包的团队，应重新构建 MCP server 镜像，轮换 server 可访问的 API token 与仓库凭据，清理本地/CI npm 缓存，并把 MCP server 依赖纳入 SCA、lockfile 审计和安装脚本阻断策略。对代码 Agent，应在工具调用前增加包来源校验和敏感工具二次审批。

**信息来源**：
- https://www.aikido.dev/blog/red-hat-npm-packages-compromised-credential-stealing-worm

## HP Poly VoIP RCE 把传统嵌入式漏洞变成 AI 语音欺诈数据入口

**入选理由**：Rapid7 在 CVE-2026-0826 技术分析外同步发布风险解读，明确指出 VoIP 电话被攻陷后可成为高质量语音数据采集点，支撑 vishing、deepfake 和伪造授权等 AI 语音欺诈。

**技术细节**：漏洞本身是 SIP/SDP 解析路径上的栈溢出，攻击者可通过 UDP 5060 未认证请求在 HP Poly VVX/Trio 设备上获得 root 权限。Rapid7 指出，电话设备常位于高价值会议室、管理层办公室、服务台、交易台或医疗站点，且通常不部署现代 EDR。攻击者一旦持久控制这类设备，除传统监听外，还可收集干净的目标说话人音频，用于合成语音、社工验证绕过或财务授权欺诈。

**影响与处置**：安全团队应把语音设备从“办公外设”重新建模为带麦克风、联网能力和管理逻辑的计算资产。处置上除固件升级、语音 VLAN 隔离、SIP 流量限制外，还应为高价值会议空间建立异常通话、异常设备重启、配置变更和外联监控；涉及高管或财务审批的语音确认流程，应增加多因素或带外验证，避免单独依赖声音相似性。

- https://www.rapid7.com/blog/post/ve-cve-2026-0826-how-an-old-bug-can-feed-ai-powered-impersonation
