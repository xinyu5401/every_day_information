## OpenAI 将 ChatGPT 记忆系统升级为基于 Dreaming 的新架构

**入选理由**  
OpenAI 官方于 2026-06-04 发布研究/产品说明，涉及长期上下文记忆架构、可用人群与推理成本变化，属于会直接影响产品行为与评测口径的能力更新。

**技术细节**  
新版记忆系统从早期的 saved memories，演进到以 `Dreaming V3` 为核心的后台记忆合成架构。官方说明其会从多轮历史对话中自动整理可持续使用的用户上下文，而不是只依赖显式“记住这件事”的触发；用户可以通过 memory summary 查看、修正或删除总结出的记忆。OpenAI 同时给出两项关键工程信号：一是该架构强调“更强且更省算力”，二是面向免费用户的推理成本较此前下降约 `5x`，因此开始向 Free 用户逐步开放，并提高 Plus/Pro 用户的记忆容量。

**影响与处置**  
这类更新会直接影响个性化评测、历史上下文泄露面、长期偏好漂移和红队测试边界。对接入 ChatGPT 作为工作流前端的团队，建议重新验证三类场景：跨会话隐式记忆写入、过期偏好是否被自动遗忘、以及敏感项目上下文能否被用户侧可见摘要完整追踪和纠偏。

## Code2LoRA 用超网络按仓库生成 LoRA，试图替代长上下文仓库注入

**入选理由**  
该论文在 Hugging Face Daily Papers 2026-06-05 榜单居前，且对象明确、指标充分，讨论的是代码模型在仓库级上下文和持续演化代码库中的参数高效适配问题。

**技术细节**  
`Code2LoRA` 不是把仓库知识继续塞进更长 prompt，而是让超网络直接生成“仓库专属 LoRA 适配器”，从而实现 `zero inference-time token overhead`。论文分为两个场景：`Code2LoRA-Static` 针对静态仓库快照；`Code2LoRA-Evo` 用 GRU 隐状态按 commit diff 持续刷新适配器。作者同步构建了 `RepoPeftBench`，覆盖 `604` 个 Python 仓库，其中静态轨道含 `40K` 训练与 `12K` 测试 assertion-completion 任务，演化轨道含 `215K` 训练与 `87K` 测试任务。结果上，Static 轨道达到 `63.8%` cross-repo 与 `66.2%` in-repo exact match；Evo 轨道达到 `60.3%` cross-repo EM，相比单一共享 LoRA 提升 `5.2` 个百分点。

**影响与处置**  
对代码 Agent、仓库问答和私有代码补全来说，这条路线的价值在于把“仓库知识注入”从 prompt 工程迁移到参数侧，理论上可以同时减少上下文占用与检索延迟。需要继续观察的是：大重构、API 语义漂移和噪声 diff 条件下，GRU 驱动的适配器刷新是否会累积偏差。
