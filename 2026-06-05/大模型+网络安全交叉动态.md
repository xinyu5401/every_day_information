# 大模型+网络安全交叉动态日报（2026-06-05）

检索窗口：重点覆盖 2026-06-04 至 2026-06-05（北京时间），并额外复核 Agent/MCP、工具调用审批、自动化编码工具、开发者供应链与 AI 安全红队相关材料。

## Red Hat npm/Miasma：GitHub Actions OIDC trusted publishing 链路被用于投递凭据窃取蠕虫

**入选理由**：多家安全团队在过去数日持续发布技术分析，Firecrawl 开发者供应链固定关键词命中 Red Hat `@redhat-cloud-services` npm 包、GitHub Actions OIDC trusted publishing、`id-token: write`、preinstall 钩子和 MCP 相关包名。事件同时影响开发者 CI/CD、包仓库发布链路和 Agent/MCP 工具生态，按归类优先级进入交叉领域。

**技术细节**：攻击者污染多个 `@redhat-cloud-services` npm 包版本，恶意逻辑通过 `preinstall` 钩子在 `npm install` 阶段执行，收集 AWS、Azure、GCP、Vault、Kubernetes、GitHub Actions 等凭据与运行环境信息。Microsoft 将该活动命名为 Miasma，指出超过 90 个包版本受到影响；Aikido、StepSecurity、Orca、Mend、SafeDep 等分析均指向 `@redhat-cloud-services` 命名空间被投递凭据窃取载荷。SafeDep 的分析进一步指出攻击者滥用了 GitHub Actions OIDC trusted publisher 发布链路，涉及短期发布令牌获取路径，而不是传统长期 npm token 泄露模型。

**Agent/MCP 安全补齐字段**：命中包名中包含 MCP 相关组件时，应将其视为代码 Agent/MCP 工具供应链风险，而不只是普通 npm 恶意包。需要检查 Agent 工具入口、默认可调用工具、CI runner 中挂载的云凭据、Kubernetes/Vault/GitHub token、`permissions: id-token: write` 使用位置、workflow 触发条件、包锁文件与私有镜像缓存，以及工具调用日志中是否能追踪“外部输入 -> npm install -> preinstall -> 凭据读取/外传”的传播路径。撤回恶意包后，仍需排查 lockfile、缓存、私有 registry 镜像和构建产物中的滞留版本。

**影响与处置**：短期动作是冻结受影响范围内的自动构建，定位安装过相关版本的开发机、CI runner 和容器镜像，轮换云账号、Vault、Kubernetes、GitHub、npm token，并审计 GitHub Actions `permissions` 最小化配置。中期应为 trusted publishing 增加发布主体约束、provenance/attestation 校验、包名与发布 workflow 绑定关系审计，并对自动化编码/Agent 工具默认执行 `npm install` 的场景增加人工审批和网络隔离。

## Microsoft：更新 Agentic AI 失败模式 taxonomy，强调工具链与间接注入风险

**入选理由**：Microsoft Security 于 2026-06-04 发布 Agentic AI 系统失败模式复盘，属于高质量一手红队经验总结；内容命中 GUI/Agent 安全、工具攻击、间接 prompt injection、工具审批和日志审计等固定监测项。

**技术细节**：该文基于一年 Agentic AI 红队经验，重点讨论模型从“回答文本”转向“调用工具、读写外部状态、执行多步任务”后新增的失败模式。核心风险包括外部内容注入进入工具参数、长期记忆或上下文污染、权限边界不清、工具输出被重新注入模型上下文、审批提示疲劳，以及审计字段不足导致难以还原攻击链。

**影响与处置**：企业上线代码 Agent、IDE Agent、MCP server 或自动化运维 Agent 时，应把工具清单、资源权限、凭据挂载、审批结果、工具参数、外部内容来源、命令执行结果和回滚动作写入统一审计。对高风险工具调用应采用 capability allowlist、参数 schema 校验、最小权限 token、隔离执行环境和可撤销会话，而不是只依赖一次性“允许/拒绝”弹窗。

**信息来源**：
- https://www.microsoft.com/en-us/security/blog/2026/06/04/updating-taxonomy-failure-modes-agentic-ai-systems-year-red-teaming-taught-us/

## Figma Make：上线工具调用审批，设计 Agent 开始显式暴露高风险执行边界

**入选理由**：Figma 2026-06-03 release notes 提到 Make 中的 tool-call approvals，虽然不是漏洞事件，但属于 Agent 工具审批机制的头部产品实质更新，命中固定关键词 `tool approval` 与 `tool guardrail`。

**技术细节**：该更新允许用户在内容进入会话前审查工具调用，实质上把 Agent 对外部工具/上下文的写入边界显式暴露给用户。对设计和前端生成场景，风险点主要包括外部资源读取、生成代码或组件配置写入、插件/工具输出再注入，以及团队文件中的敏感信息被跨上下文使用。

**影响与处置**：采用 Figma Make 或类似设计 Agent 的团队，应把审批事件纳入团队审计，记录工具名称、参数、目标文件、外部 URL、执行结果和操作者。若工具可生成代码或同步到仓库，还需和代码评审、依赖扫描、密钥扫描联动。

**信息来源**：
- https://www.figma.com/release-notes/

## agentgateway 加入 AAIF：Agentic AI 网关路线继续向协议治理收敛

**入选理由**：Solo.io 于 2026-06-04 发布 agentgateway 加入 Agentic AI Foundation（AAIF）的消息；该项目定位为 Agentic AI infrastructure 的开放网关，涉及 MCP/A2A/API 访问治理，属于 Agent/MCP 运行时安全观察点。

**技术细节**：agentgateway 面向 Agent 与外部工具/服务之间的网关层，关注协议桥接、策略执行、可观测性和流量控制。其安全价值不在“阻止所有 prompt injection”，而是在工具调用前后提供集中策略点：认证、授权、审计、限流、协议转换和工具目录治理。

**影响与处置**：企业如果已经使用 MCP server、A2A 或自研工具调用网关，应关注该类项目是否能提供标准化审计字段、按工具/租户/资源粒度授权、敏感参数脱敏、调用回放和隔离策略。短期不建议把网关等同于内容安全模型，仍需在模型输入、工具参数和执行环境三层做校验。

**信息来源**：
- https://www.solo.io/blog/agentgateway-joins-aaif-as-an-open-gateway-for-agentic-ai-infrastructure
- https://github.com/agentgateway/agentgateway
