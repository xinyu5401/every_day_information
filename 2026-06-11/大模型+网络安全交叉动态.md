# 大模型+网络安全交叉动态（2026-06-11）

## 1. LiteLLM `CVE-2026-42271`：MCP stdio 测试端点可被低权限 API key 直接拉起主机子进程

**入选理由**

这是典型的“大模型基础设施 + MCP / Agent 工具入口 + 已进 CISA KEV”的交叉事件，不应被当作普通 Web 命令注入条目处理。它直接打在 LLM 网关的 MCP 测试能力上，且利用前提只需有效 proxy API key。

**技术细节**

- 漏洞：`CVE-2026-42271` / `GHSA-v4p8-mg3p-g94g`
- 组件：`pip litellm`
- 受影响版本：`>= 1.74.2, < 1.83.7`
- 修复版本：`1.83.7`
- 漏洞入口：
  - `POST /mcp-rest/test/connection`
  - `POST /mcp-rest/test/tools/list`
- 触发链：
  - 两个端点原本用于“保存前测试 MCP server”；
  - 请求体可直接提交完整 server 配置，包括 stdio transport 的 `command`、`args`、`env`；
  - 服务端测试连接时会在代理主机上以 proxy 进程权限拉起对应子进程。
- 权限边界缺陷：
  - 漏洞前仅要求“有效 proxy API key”；
  - 未要求 `PROXY_ADMIN`；
  - 因此低权限 internal-user key 也可触发主机命令执行。
- CISA KEV：
  - `Date Added`: **2026-06-08**
  - `Due Date`: **2026-06-22**
  - 今日窗口判断：**未命中**

**影响与处置**

- 这是代码 Agent / MCP 基础设施中的“工具入口即命令执行”问题，风险不只在 LiteLLM 自身，还在于代理宿主机可访问的密钥、模型凭据、企业数据平面和下游工具链。
- 建议动作：
  - 立即升级到 `1.83.7` 或更高；
  - 若暂不能升级，先在反向代理或 API Gateway 阻断上述两个测试端点；
  - 复查所有 MCP server 配置与测试日志，重点看是否出现异常 `command` / `args` / `env`；
  - 对代码 Agent 基线补齐：可调用工具白名单、MCP 测试权限审批、工具调用审计字段、宿主机密钥隔离。

**信息来源**

- https://github.com/advisories/GHSA-v4p8-mg3p-g94g
- https://www.cisa.gov/known-exploited-vulnerabilities-catalog
- https://nvd.nist.gov/vuln/detail/CVE-2026-42271

## 2. OpenSSF 复盘 Mini Shai-Hulud：`pull_request_target` + cache poisoning + `id-token: write` 才是“可信发布签名仍失守”的根因链

**入选理由**

今天的新内容不是“又一起旧新闻转载”，而是 OpenSSF 对 2026 年上半年最具代表性的开发者工具供应链攻击给出结构化技术复盘，直接点名 GitHub Actions、OIDC trusted publishing 与 SLSA 边界问题，且涉及 `@mistralai` 等 AI 厂商命名空间。

**技术细节**

- 发布时间：**2026-06-10**
- 公开影响面：文章回顾称攻击先在合法 TanStack 身份下发布 **84 个恶意包**，随后扩散到 **170+** 个 `@mistralai`、`@uipath` 等命名空间包。
- 攻击链三段式：
  - `pull_request_target` 让 fork 代码在 base repo 安全上下文内运行；
  - 恶意代码污染 pnpm cache，后续 `release.yml` 恢复该缓存；
  - 运行中的 attacker-controlled code 从 runner 内存提取 ambient OIDC token，在 `id-token: write` 权限下绕过正常发布逻辑，直接走 npm trusted publishing。
- 关键判断：
  - 恶意包携带的 provenance / attestation 在密码学上仍然有效，因为攻击者拿到的就是合法 OIDC 身份；
  - 问题不在“签名是否存在”，而在 build platform 是否具备 Build L3 级别的隔离能力。
- 防守要点：
  - 移除或严格约束 `pull_request_target` 执行 fork 代码；
  - 最小化 workflow `permissions`，特别是 `id-token: write`；
  - 按 digest 固定 action 版本；
  - 把 shared cache 视为跨 workflow 信任边界；
  - 监测“失败 workflow 却产生 publish 事件”的异常。

**影响与处置**

- 对 AI / Agent 生态，影响不只在前端依赖污染；凡是通过 CI/CD 直发 CLI、IDE 扩展、MCP 工具、SDK 或代码 Agent 组件的组织，都要重新审查 OIDC 发布链与缓存边界。
- 建议动作：
  - 盘点所有 `pull_request_target`、缓存恢复点、`id-token: write` 使用点；
  - 对 trusted publishing 增加 provenance 校验之外的运行时监测；
  - 若产物服务于代码 Agent / IDE Agent / MCP server，再额外核查默认权限、凭据挂载和自动更新链路。

**信息来源**

- https://openssf.org/blog/2026/06/10/mini-shai-hulud-where-slsas-boundaries-fall/

## 3. 新论文《Toward Secure LLM Agents》发布：把 Agent 安全统一为“信息流 + 委托权限 + 持久状态”问题

**入选理由**

该论文为 **2026-06-09** 新上架的 `arXiv:2606.10749`，不是单点 PoC，而是对 247 篇 LLM Agent 安全文献做系统归纳；对日报关注的 GUI agent、工具调用攻击、MCP、长期记忆污染和多 Agent 传播都有直接覆盖。

**技术细节**

- 论文：`Toward Secure LLM Agents: Threat Surfaces, Attacks, Defenses, and Evaluation`
- arXiv：`2606.10749v1 [cs.CR]`
- 方法：
  - 汇总 **247** 篇论文；
  - 用生命周期视角统一输入、规划、决策、工具执行、输出、记忆、监控、协同八个环节。
- 主要结论：
  - 当前研究仍以 prompt injection 与 tool-mediated control-flow hijacking 为主；
  - 但**持久状态污染**与**多 Agent 传播**正在成为中心风险；
  - 现有 benchmark 仍明显低估长时程、有状态、真实部署环境下的风险。
- 文中给出的高频威胁面：
  - `Web Content`：55 篇
  - `Tool Outputs`：54 篇
  - `Retrieved Content`：37 篇
  - `Memory / Scratchpads`：25 篇
- 对工程基线的启发：
  - 工具执行前需要显式 capability checks；
  - 状态管理要具备 provenance-aware isolation、revocation / decay；
  - 多 Agent 场景要把 coordination 当成独立安全语义，而不是普通消息传递。

**影响与处置**

- 对企业内部 Agent 平台，这篇综述的价值在于它把“提示词安全”扩展成“运行时系统安全”，适合作为后续安全基线、评测表和审计字段的设计参考。
- 若正在建设 MCP / Tool-use / Coding Agent，建议优先补齐：
  - 工具权限边界；
  - 长期记忆污染与撤销机制；
  - 外部内容到工具参数的传播观测；
  - 多 Agent 委托与消息传播审计。

**信息来源**

- https://arxiv.org/abs/2606.10749
- https://arxiv.org/pdf/2606.10749
