# 2026-06-13 大模型+网络安全交叉动态

## 1. PI-Hunter 把 prompt injection 审计从“打中一次”推进到“暴露来源与传播路径”

**入选理由**  
这是过去几天里最值得持续跟踪的 Agent 安全论文之一，且今天仍在 AI Security 研究传播面中升温，命中固定关键词 `tool attack`、`GUI agent security`、`OpenClaw`、`tool guardrail`。

**技术细节**  
PI-Hunter 不是单纯优化攻击成功率，而是做 source-aware test case 构造、反馈驱动的演化探索和 trajectory-level auditing，用来暴露“隐藏注入指令是如何被检索、信任、传播并执行”的完整链路。论文在多个 benchmark、agent 架构、攻击类型和现有防护方案上评估，宣称相对强基线显著提高 vulnerability exposure 与 attack-surface coverage；在 PDF 公开结果中，还覆盖了 Spotlight、MELON、PIGuard 等 defense 条件。论文明确将未来扩展目标指向更复杂的生产级 agent，例如 OpenClaw。

**影响与处置**  
对代码 Agent、MCP server、检索型企业助手而言，红队框架应从“单轮 jailbreak”升级到“源定位 + 指令定位 + 轨迹审计”。落地上至少要补齐：工具入口、外部内容到工具参数的传播路径、工具调用审批点、日志中的 source ID / tool args / tool result / final action 关联字段。

**信息来源**  
- arXiv: https://arxiv.org/abs/2606.12737  
- PDF: https://arxiv.org/pdf/2606.12737

## 2. 新基准把 Web Agent prompt injection 风险从攻击视角改写为“受害者视角”

**入选理由**  
`Who Pays the Price?` 不是又一个 attack benchmark，而是把用户、卖家、平台等不同受害方拆开评估，更贴近真实代理执行后果，属于 Agent 执行安全的高价值观察对象。

**技术细节**  
论文在真实 Web agent 环境中构造 stakeholder-centric benchmark，区分受影响实体，分解攻击目标，并同时使用 outcome-level 与 process-level 指标。作者指出，目前 agent 对任何单一攻击目标都没有稳定抵抗力，并总结出几类不同失败模式：`stealthy parasitism`、`misaligned disruption`、`compounded failure`。这意味着只看“任务是否完成”会低估攻击成功面，因为有些攻击能在不打断用户委托任务时悄悄达成对抗目标。

**影响与处置**  
做浏览器 Agent、购物/办公代理、自动下单或多方交易流程的团队，需要把“谁承担损失”单列进评测与审计，而不是只统计 overall success rate。审批策略应覆盖高后果动作，例如下单、转账、发送邮件、改写工单和提交代码。

**信息来源**  
- arXiv: https://arxiv.org/abs/2606.13385  
- 基准仓库链接见论文页: https://arxiv.org/abs/2606.13385

## 3. RAG poisoning 研究给出 432 组组合实验，指出检索深度和多库复制会系统性放大污染

**入选理由**  
该工作给出的不是单点攻击演示，而是面向实际 RAG 参数面的大样本因子分析，适合直接转成工程侧配置基线。

**技术细节**  
论文对数据集、retriever 类型、retrieval depth、数据库组成、chunking 策略、generator model 做了 432 组全因子实验。结论上，`retriever architecture`、`dataset` 与 `retrieval depth` 对 poisoning 暴露最敏感；dense 与 graph-based retriever 相比 BM25 通常更稳健，而更大的 retrieval depth 会增加把 poisoned passage 拉进上下文的概率。论文还指出，多数据库同时复制有毒内容会放大攻击影响，额外引入干净来源则有缓解作用。

**影响与处置**  
RAG 安全基线不应只盯模型层。短期可执行动作包括：限制默认检索深度、减少同一内容在多库/多索引中的复制扩散、在索引构建前做内容来源分级与 poisoning 扫描，并把 retrieval trace 纳入审计日志。

**信息来源**  
- arXiv: https://arxiv.org/abs/2606.12469

## 4. “只改表述不改证据”也能操纵 AI 评审，提示评审 Agent 已出现新的优化面

**入选理由**  
这不是传统 prompt injection，但属于对 LLM 决策流程的现实攻击面扩展，适合作为“间接操控 Agent”的边缘热点观察。

**技术细节**  
论文研究的是 `presentation-only revisions`：不改实验、图表、公式、证明与数值结果，只改摘要、贡献 framing、related work、讨论和叙事结构。作者在三个主流 AI reviewer 上做闭环优化，得到 75.1% 的攻击成功率与平均 `+1.21/10` 评分提升；效果主要来自改变评审器如何解释论文，而不是常规文字润色。

**影响与处置**  
凡是把 LLM 用于审稿、风控、简历筛选、申报评估或工单优先级判定的系统，都应假设“表述优化攻击”会出现。缓解方向是把模型判断锚定到结构化证据字段，并对 narrative-only 变更单独做差异审查。

**信息来源**  
- arXiv: https://arxiv.org/abs/2606.13044
