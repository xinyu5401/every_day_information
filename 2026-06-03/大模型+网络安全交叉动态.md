# 大模型 + 网络安全交叉动态日报（2026-06-03）

## OpenAI Codex 扩展 role-specific plugins、Sites 与 annotations，Agent 权限边界需要重新审计

**入选理由：** OpenAI 于 2026 年 6 月 2 日发布 Codex 产品更新，新增面向数据分析、创意生产、销售、产品设计、公开股票投资、投行等角色的插件，并推出 Sites 预览和 annotations。该事件涉及代码/知识工作 Agent 的工具权限、第三方应用连接、工作区共享 URL 与内容生成后的二次编辑链路，归入 Agent 执行安全。

**技术细节：**

- 插件范围：6 类 role-specific plugins，公告称合计包含 62 个 popular apps 和 110 个 skills；覆盖 Snowflake、Databricks、Hex、Tableau、Figma、Canva、Salesforce、HubSpot、Slack、FactSet、LSEG、PitchBook 等外部系统。
- Sites：Business 与 Enterprise 客户预览可用；Codex 可生成交互式 hosted websites/apps，并通过 URL 在 workspace 内共享；可用于 dashboards、planners、review workspaces、project boards、lightweight tools 等。
- annotations：允许用户指向 Codex 创建的代码、Markdown、网站、文档、表格、幻灯片中的具体区域并要求局部修改。
- 管理控制：OpenAI 称 Business/Enterprise workspace 管理员可在 workspace settings 控制 underlying app permissions；Enterprise admins 可在 admin settings 启用 Sites。

**影响与处置：** 插件把 Codex 的可访问资源从代码仓库扩展到企业 SaaS、数据仓库、CRM、设计工具和财务数据。安全基线应补齐：插件安装审批、每个 app 的 OAuth scope、默认可调用工具清单、Sites 的 URL 可见范围、生成站点是否可读取敏感上下文、annotations 是否会把外部内容重新注入工具参数、Codex 任务日志是否记录工具名/参数/返回摘要/审批人/执行结果。对销售、金融、数据分析插件，应额外审计客户数据、MNPI、凭据、BI 查询和外部图片/素材生成链路。

**信息来源：**

- OpenAI 产品公告：https://openai.com/index/codex-for-every-role-tool-workflow/
- OpenAI Codex 知识工作报告页：https://openai.com/index/codex-for-knowledge-work/
- Codex Sites 文档：https://developers.openai.com/codex/sites#get-started-with-sites

## AgentRedBench：215 个 SaaS 集成场景显示工具响应层间接提示注入仍是生产级风险

**入选理由：** arXiv 2026 年 6 月新论文《AgentRedBench: Dynamic Redteaming and Integration-Aware Defense for LLM Agents over SaaS Integrations》命中 GUI/工具调用/Agent 安全固定监测方向，并给出跨厂商模型面板、动态攻击者和 guard 评测。

**技术细节：**

- 攻击面：工具调用 Agent 读取 Gmail、Salesforce、Jira、Slack 等第三方 SaaS 集成返回内容；攻击者只控制集成中的响应内容，不控制模型权重、工具 schema 或 agent 代码。
- 数据集：215 个 canonical scenarios，覆盖 24 个企业集成、9 个功能家族、5 类攻击类型；包含 multi-integration chained scenarios。
- 攻击类型：论文附录列出 output_channel_url_relay、destination_hijack、content_hijack、tool_argument_hijack、tool_family_creep 等。
- 评测结果：8 模型面板覆盖 Anthropic、OpenAI、Google；无 guard 的 ASR 范围从 Claude Sonnet 4.6 的 32% 到 Gemini 3 Flash 的 81%。
- 防护：论文开放 Agentredguard，定位为 tool-response content 上的 inference-time guard；报告称可将面板 ASR 从 69.9% 降至 2.4%，false-positive rate 0.37%，median CPU latency 9.5 ms。canonical scenarios 通过 maintainer-mediated channel 评测，以降低训练集污染。

**影响与处置：** 企业 Agent 的风险不只来自用户输入，而是来自“读一个系统、写另一个系统”的跨集成动作。上线 SaaS Agent 时应把 tool response 作为不可信输入，记录每次工具调用的 integration、resource id、原始返回摘要、guard 判定、目标 action、目标参数和用户授权边界；对跨系统写操作启用 step-up approval；对 URL、收件人、金额、仓库、ticket assignee、CRM 字段变更等高风险参数做 policy diff。

**信息来源：**

- arXiv HTML：https://arxiv.org/html/2606.02240v1
- arXiv PDF：https://arxiv.org/pdf/2606.02240

## SeClaw：面向 OpenClaw/自主 Agent 的规范驱动安全任务合成与轨迹审计框架

**入选理由：** arXiv 2026 年 6 月论文《SeClaw: Spec-Driven Security Task Synthesis for Evaluating Autonomous Agents》命中 OpenClaw、MCP server、tool attack、agent tracing 固定关键词，关注点从最终回答扩展到执行轨迹与工具环境。

**技术细节：**

- 方法：Spec-Driven Security Task Synthesis 从结构化风险规格生成安全任务，规格包含 risk sources、deployment scenarios、threat methods、tool-use requirements。
- 环境：SeClaw docker 提供隔离执行环境，模拟 multi-turn agent-user interactions；通过 ToolHub-based Skill 和 MCP 配置重建任务环境。
- 风险覆盖：资源风险、用户任务风险、环境风险、Agent intrinsic behavior 风险；涉及 system instructions、MCP servers、skills、memory、external capabilities、prompt injection、malicious skill injection、memory poisoning、privilege abuse 等。
- 审计：框架记录标准化 agent-environment trajectories，用于复现、归因、比较和失败诊断，而不是只看最终响应是否违规。
- 代码：论文称代码可用地址为 https://github.com/seclaw-eval/seclaw-eval。

**影响与处置：** SeClaw 的价值在于把 Agent 安全评测从“手工红队样例”转向可扩展任务合成与轨迹级审计。对代码 Agent、MCP/插件平台和自动化办公 Agent，建议将评测用例拆成：可访问资源、工具参数、凭据挂载、外部内容注入点、长期记忆污染、命令执行权限、审批策略、日志字段、回滚机制，并在 CI 中对工具调用轨迹做回归测试。

**信息来源：**

- arXiv HTML：https://arxiv.org/html/2606.02302v1
- arXiv PDF：https://arxiv.org/pdf/2606.02302v1

## CISA KEV 中 Nx Console 与 TanStack 供应链条目仍在处置窗口，涉及 IDE 扩展、OIDC trusted publishing 与凭据外泄

**入选理由：** CISA KEV 仍将 CVE-2026-48027（Nx Console）与 CVE-2026-45321（TanStack）列为已知在野利用，Due Date 均为 2026-06-10。两者均属于开发者工具供应链，且 Nx 事件由 TanStack 供应链泄露 GitHub 凭据后进一步影响 VS Code/Cursor 扩展发布链。

**技术细节：**

- CVE-2026-48027 / Nx Console：GHSA 披露恶意 Nx Console 18.95.0 于 2026-05-19 12:30 UTC 发布到 Visual Studio Marketplace，约 18 分钟后移除；OpenVSX 暴露约 36 分钟。NVD CVSS 3.1 为 9.8 Critical。恶意扩展拉取混淆 payload，采集 Vault、npm、AWS、GitHub、1Password、文件系统私钥、GCP/Docker 凭据等，并通过 HTTPS、GitHub API、DNS 外传；Linux 上还尝试 sudoers 持久化。修复版本为 18.100.0。
- CVE-2026-45321 / TanStack：GHSA 披露 2026-05-11 19:20-19:26 UTC 间，42 个 `@tanstack/*` 包共 84 个恶意版本发布到 npm。攻击链包含 `pull_request_target` 配置问题、GitHub Actions cache poisoning、Actions runner 进程内 OIDC token 提取，并通过合法 GitHub Actions OIDC trusted-publisher 绑定发布。NVD CVSS 3.1 为 9.6 Critical。
- TanStack payload：安装时执行约 2.3 MB 混淆 `router_init.js`，采集 AWS/GCP/Kubernetes/Vault/npm/GitHub/SSH 私钥等；通过 Session/Oxen messenger 文件上传网络外传；还会枚举受害者维护的 npm 包并尝试再发布，形成传播。
- 关键 IOC：TanStack 恶意 optional dependency 包含 `@tanstack/setup` 指向 `github:tanstack/router#79ac49eedf774dd4b0cfa308722bc463cfe5885c` 的 orphan commit；Nx IOC 包括 `cat.py`、`com.user.kitty-monitor.plist`、`.gh_update_state`、`__DAEMONIZED=1` 进程等。

**影响与处置：** 这两起事件说明 OIDC trusted publishing 不等于无风险，短期 token 可被 runner 侧内存提取并用于“可信身份”发布。处置优先级：检查 2026-05-11 后 CI 是否安装过受影响 `@tanstack/*` 版本，检查 2026-05-19 暴露窗口是否安装 Nx Console 18.95.0；删除锁文件和 node_modules 后干净重装；轮换所有可从开发机/CI runner 读取的 npm、GitHub、SSH、云、Vault、Kubernetes 凭据；审计 `pull_request_target`、cache restore/save 跨信任边界、`permissions: id-token: write`、trusted publisher provenance、孤儿 commit git URL 依赖和私有镜像缓存。

- Nx Console GHSA：https://github.com/nrwl/nx-console/security/advisories/GHSA-c9j4-9m59-847w

## Langflow CVE-2025-34291 今日到期窗口命中：CORS + SameSite=None 可导致令牌获取与 RCE

**入选理由：** CISA KEV 显示 CVE-2025-34291 Date Added 为 2026-05-21，Due Date 为 2026-06-04，命中本次运行日未来 24 小时处置窗口。Langflow 属于 LLM 应用/Agent 编排平台，归入交叉领域。

**技术细节：**

- 漏洞类型：origin validation error；NVD CVSS 3.1 为 8.8 High。
- 影响版本：NVD 描述称 Langflow versions up to and including 1.6.9；CISA notes 指向 Langflow 1.9.3 release。
- 攻击条件：`allow_origins='*'` 与 `allow_credentials=True` 的宽松 CORS 配置，叠加 refresh token cookie `SameSite=None`，允许恶意网页跨源携带凭据调用 refresh endpoint。
- 后果：攻击者可获取新的 access_token / refresh_token，并访问 authenticated endpoints；结合内置代码执行能力可达任意代码执行与系统接管。

**影响与处置：** 暴露在浏览器会话、内部低信任网络或共享工作站环境的 Langflow 优先升级与复核配置。短期处置包括升级至修复版本、限制 CORS allowlist、避免 refresh token cookie 跨站发送、轮换已登录用户 token、审计近期跨源 refresh 调用和 authenticated code-execution endpoint 调用。

**信息来源：**

- CISA KEV JSON：https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json
- Langflow release：https://github.com/langflow-ai/langflow/releases/tag/v1.9.3
- NVD CVE-2025-34291：https://nvd.nist.gov/vuln/detail/CVE-2025-34291
