## 1. Red Hat npm 供应链事件命中 MCP 包：OIDC trusted publishing 与 Agent 凭据暴露面需合并处置

**入选理由**：Unit 42 2026-06-02 更新的 npm 供应链研究确认，2026-06-01 至少 32 个 `@redhat-cloud-services` 包被污染；受影响列表包含 `hcc-feo-mcp`、`hcc-kessel-mcp`、`hcc-pf-mcp` 等 MCP 相关包，触发“开发者工具供应链 + Agent/MCP 安全”升级规则。该事件虽早于 24 小时，但在今日固定关键词检查中命中 MCP、OIDC、trusted publishing、orphan commit workflow 和 developer secrets，应作为补漏热点处理。

**技术细节**：攻击者使用被入侵的 Red Hat 员工 GitHub 账号向多个 RedHatInsights 仓库推送恶意 orphan commits，绕过常规代码审查；随后触发 GitHub Actions 获取 OIDC token，并通过 npm trusted publishing 发布带有效 SLSA provenance 的木马包。payload 名为 Miasma，分析样本将正常约 200KB 的 `index.js` 替换为约 4.29MB 混淆 payload。凭据采集覆盖 GitHub token、npm token、SSH keys、AWS/GCP/Azure 凭据、Kubernetes service-account token、Vault secrets、GitHub Actions/CircleCI 等 CI/CD secrets；Unit 42 还记录了对 Claude/MCP 配置文件的读取路径，如 `~/.claude/mcp.json`、`~/.claude.json`、`~/.kiro/settings/mcp.json`。

**Agent/MCP 安全补充字段**：入口为 npm install 与 CI/CD 发布流水线；可访问资源取决于开发者机器、CI runner、MCP/Claude/Kiro 配置中挂载的 token、工具和服务器地址；命令执行权限来自安装期 JavaScript payload、Bun runtime 和 runner 环境。审批机制通常不覆盖包安装期脚本和 trusted publishing token 交换；日志字段应重点保留 workflow run、`id-token: write`、npm provenance、发布主体、orphan commit、缓存恢复、异常大文件 payload、安装期外联和 MCP 配置文件读取。

**影响与处置**：仅验证 SLSA provenance 不足以证明包内容可信，因为污染发生在受信流水线内部。建议立即冻结受影响版本，清理 lockfile、私有 registry 缓存和 CI 镜像；轮换 GitHub/npm/cloud/Kubernetes/Vault/Claude/MCP 相关密钥；审计 GitHub Actions 是否允许宽泛 `id-token: write`、`pull_request_target`、Actions cache 共享和仓库级 trusted publisher 绑定；对 MCP server 默认工具、外部输入到工具参数的传播路径和工具调用日志做额外复核。

**信息来源**：https://unit42.paloaltonetworks.com/monitoring-npm-supply-chain-attacks/

## 2. NocoDB MCP `readAttachment` 缺少所有权校验：低权限 MCP token 可跨 base/workspace 读附件

**入选理由**：GitHub Advisory Database 2026-06-05 发布/复核，漏洞直接位于 MCP 工具调用路径，虽 CVSS 较低但对多租户 MCP 权限边界有代表性。

**技术细节**：CVE-2026-47388 / GHSA-xxpj-q764-9r6q 影响 npm 包 `nocodb` `<=2026.05.0`，修复版本为 `2026.05.1`。漏洞根因是 MCP `readAttachment` 工具接受调用者提供的 `path`/`url` 并经 storage adapter 直接流式返回文件，没有验证该文件是否属于调用者 MCP context 对应的 base。修复逻辑改为在 `nc_file_references` 中查找路径，并要求存在未删除记录且 `base_id` 匹配调用者 MCP context；不匹配时返回不可访问错误。利用条件为攻击者持有低权限 MCP token 且知道附件路径。

**影响与处置**：该漏洞说明 MCP token 的最小权限不能只停留在“工具可调用”，还必须绑定租户、base、workspace 和具体资源所有权。建议升级到 `2026.05.1`，轮换长期 MCP token，检查 MCP 工具日志中的 `readAttachment` 参数、附件路径枚举、跨 workspace 访问和 shared storage 异常读取。

**信息来源**：https://github.com/advisories/GHSA-xxpj-q764-9r6q；https://github.com/nocodb/nocodb/releases/tag/2026.05.1

## 3. CUA-HandCrafted：浏览器 Agent 注入安全与代码 Agent 注入安全出现域条件差异

**入选理由**：arXiv 2026-06-05 新论文命中 GUI agent security、tool attack、agent tracing 与代码 Agent 安全基线。论文不是漏洞披露，但对红队评测可复现性和跨域泛化风险有直接价值。

**技术细节**：论文构建 CUA-HandCrafted，包含 793 个 episode、24 个多步骤 Web 任务、56 个攻击模板、8 类攻击家族和 4 种系统提示配置，用来复现近期计算机使用 Agent（CUA）红队论文中的提示注入攻击。作者报告，在 Claude Sonnet 4.6 与 GPT-5.4 上，多步骤浏览器攻击成功率为 0/140，95% Clopper-Pearson 上界 2.60%，且 prompt ablation 指向模型权重中的浏览器域加固。但同一模型权重迁移到姐妹代码 Agent 基准 SkillBench 时，手工 skill-injection 最高可达 100% 成功率。论文认为，公开文献中的高 ASR 很大程度来自 RL 优化的注入文本，而不是攻击类别本身；如果不公开优化字符串，ASR 难以复现。

**影响与处置**：不能把浏览器 Agent 的安全结果外推到代码 Agent、CLI Agent 或 MCP 工具调用面。代码 Agent 安全基线应单独记录：工具入口、可访问文件/仓库/凭据、命令执行权限、审批策略、外部内容注入点、轨迹日志、回滚机制和技能/插件安装路径；红队报告应公开足够的攻击字符串或生成方法，避免只报告不可复现的 ASR。

**信息来源**：https://arxiv.org/abs/2606.05233

## 4. Deep research Agent 评测污染：搜索时泄漏可使公开基准成绩膨胀

**入选理由**：arXiv 2026-06-05 新论文，直接命中深度研究 Agent、agent tracing 和评测可信度；对安全评估、模型榜单和企业选型有影响。

**技术细节**：论文定义 Search-Time Contamination（STC）：深度研究 Agent 在推理时主动搜索互联网，可能检索到公开基准元数据、题目上下文甚至标准答案，从而绕过原本要测的推理过程。作者划分 3 类污染：Benchmark Metadata Leakage、Question-Context Leakage、Explicit Answer Leakage，并提出检测算法量化其对成绩的影响。对现代 deep research agents 在 6 个公开基准上的评估显示，STC 普遍存在，最高可带来约 4% 的性能膨胀。

**影响与处置**：面向安全研究、渗透自动化和企业知识检索的 Agent 评测，应记录搜索轨迹、隔离测试沙箱、控制 benchmark 可见性，并区分“检索命中答案”和“模型推理得出答案”。否则公开榜单可能高估真实推理与调查能力。

**信息来源**：https://arxiv.org/abs/2606.05241

## 5. SentinelBench：长运行监控 Agent 需要同时评估完成率、反应时间与资源消耗

**入选理由**：arXiv 2026-06-05 新论文，针对长时间运行的监控型 Agent 建基准，与 SOC 告警跟踪、漏洞情报监控和自动化响应场景相关。

**技术细节**：SentinelBench 包含 100 个任务、10 个合成 Web 环境，覆盖邮件、日历、金融、职业社交、娱乐等会随时间变化的页面。任务要求 Agent 在环境变化时及时响应，而不是持续刷新或无效工具调用。指标包括任务完成率、反应时间和资源使用，论文在 3 个模型与 2 个浏览器 Agent harness 上建立基线，显示不同 Agent 设计会显著影响响应速度与成本之间的权衡。

**影响与处置**：安全运营中的 Agent 不应只评估单次任务成功率，还要设定轮询频率、等待策略、工具调用预算和告警延迟 SLA。落地时建议保留时间戳、页面状态版本、触发事件、工具调用次数、等待原因和人工接管点。

**信息来源**：https://arxiv.org/abs/2606.05342
