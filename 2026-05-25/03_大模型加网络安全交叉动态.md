# 2026-05-25 大模型 + 网络安全交叉动态

## 摘要

交叉领域今日主线非常明确：LLM agent 的工具调用、技能生态、MCP 认证、长上下文守护栏和数据投毒正在成为新的安全边界。传统“模型回答是否违规”的评测正在扩展为“模型如何选择工具、如何解释权限、如何处理记忆和外部状态”的系统级评测。

## API / MCP 技术准确性要点

基于 Context7 对 OpenAI API Reference 的查询，当前工具调用审计至少应覆盖以下字段或等价信息：

- `tool_choice`：模型是否自动、必须、禁止或被强制调用某个工具。
- `tools` / `allowed_tools`：可用工具集合、MCP server 标签、只读过滤和工具名白名单。
- `function_call.arguments` / `mcp_call.arguments`：模型生成的 JSON 参数。
- `call_id` / `approval_request_id`：一次工具调用与其输出、审批请求之间的关联。
- `mcp_approval_request` / `mcp_approval_response`：人类审批是否发生、是否通过、拒绝理由。
- `output` / `error`：工具执行结果、错误和返回内容，需作为后续模型输入污染面处理。

## 风险场景

- **间接提示注入到工具调用**：网页、README、issue、日志、PDF 中的恶意指令诱导 agent 调用浏览器、shell、RMM 或云 API。
- **技能库供应链投毒**：恶意技能在 YAML frontmatter 中伪装成防守 playbook，正文或脚本中隐藏危险命令。
- **MCP server 认证缺陷**：远程 MCP server 若缺少 OAuth 绑定、工具白名单和审批策略，可能被越权调用。
- **记忆投毒**：恶意长期记忆让 agent 在后续任务中错误归因、信任伪造资产或绕过审批。
- **公开漏洞 + LLM 自动化**：Ghost CMS、cPanel、RMM、Linux 提权等漏洞公开细节可被快速转化为资产扫描、验证脚本和组合利用建议。

## 持续监测建议

- 建立“LLM 架构变化”监测项：重点跟踪 diffusion LLM、混合推理模型、长上下文模型对现有安全评测的影响。
- 对内部代理平台强制落地工具调用审计：记录模型输入、工具选择、工具参数、执行结果、权限来源和用户确认状态。
- 将 AI 生成钓鱼纳入 SOC 用例：使用域名、证书、页面结构、下载物、终端行为和文本相似度联合聚类。
- 为高危漏洞增加“LLM 可利用性”评分：评估公开信息是否足够让模型生成复现步骤、检测脚本或利用链。
- 跟踪 MCP server 与插件供应链：对第三方插件、MCP server、OAuth app、依赖包进行准入审查和版本锁定。
- 对高速模型 API 建立滥用风控：监测异常调用量、批量模板生成、多语言社工文案和短时间大量相似输出。
- 每日固定复核 CNVD、CNCERT、FreeBuf、奇安信、深信服、绿盟科技；若官方站点未给出精确时分，报告中继续标注“待核实”。
- 新增：对 agent 技能库建立签名与 SBOM 机制，记录技能来源、版本、作者、哈希、外链、脚本权限和测试结果。
- 新增：对 reasoning trace / agent trace 训练数据执行敏感信息扫描与危险行为标签化，避免把真实凭据、内部路径和攻击链步骤直接蒸馏进模型。
