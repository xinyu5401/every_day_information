### 3. MCP 接口层面的安全审计点明确化

- 来源：[MCP tools specification](https://github.com/modelcontextprotocol/modelcontextprotocol/blob/main/docs/specification/2025-11-25/server/tools.mdx)、[MCP schema](https://github.com/modelcontextprotocol/modelcontextprotocol/blob/main/docs/specification/2025-11-25/schema.mdx)
- Context7 查询要点：MCP 通过 `tools/list` 暴露工具定义，通过 `tools/call` 执行工具；工具定义包含 `name`、`description`、`inputSchema`、可选 `outputSchema` 等结构。
- 风险面：工具描述可影响模型选择；`inputSchema` 约束不足会扩大参数注入空间；`tools/call` 的 `name` 与 `arguments` 是审计最小粒度；若缺少用户确认、权限来源和执行结果记录，事后溯源会非常困难。
- 建议：对所有内部 agent 平台强制记录模型输入、候选工具列表、最终工具选择、工具参数、执行结果、权限来源、用户确认状态和调用链 ID。

## 影响评估

- 对研发平台：插件和 MCP server 正成为 agent 的“执行面”，需要按软件供应链标准准入。
- 对安全团队：SOC 需要从检测“模型输出”扩展到检测“模型驱动的工具行为”。
- 对评测团队：传统 prompt injection、越狱、敏感信息泄露评测不足以覆盖工具调用型 agent，需要加入浏览器、终端、代码仓库、CI/CD、云控制台等真实动作面。

## 后续建议

- 建立 MCP server 与插件供应链准入：登记来源、版本、权限、依赖包、OAuth app、更新策略和停用路径。
- 对工具调用落地强制审计：记录 `tools/list` 响应、`tools/call` 参数、执行结果、权限来源和用户确认状态。
- 为智能体应用新增“越权动作”测试：覆盖文件读写、浏览器操作、终端执行、凭据访问、外联请求和跨租户数据访问。
- 将 AI agent skills 仓库纳入供应链监测：关注新增 skill 是否引入危险命令模板、凭据收集逻辑或不受控外联。
