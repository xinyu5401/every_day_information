# 大模型+网络安全交叉动态

说明：交叉领域今日最强信号主要来自最近一批 arXiv Agent 安全论文。考虑到周末索引节奏，以下条目以 `2026-06-04` 至 `2026-06-05` 最近公开批次中仍在升温、且与 MCP/Agent/tool attack 直接相关的论文为主。

## 真实世界 MCP Server 出现“描述-代码不一致”，测得 9.93% description-code pair 存在偏差

**入选理由**  
题目直接命中固定关键词 `MCP server prompt injection` 与 `MCP authorization` 相邻风险面，不是概念讨论，而是实测真实 MCP 生态。

**技术细节**  
论文 `Description-Code Inconsistency in Real-world MCP Servers` 把问题定义为 `Description-Code Inconsistency (DCI)`：工具描述宣称的能力、边界和副作用，与真实代码实现不一致。作者提出 `DCIChecker`，把结构化静态分析与 `Direct-Reverse-Arbitration` 提示方法结合，用于交叉验证描述和实现，并在 `2,214` 个真实 MCP server 中抽取 `19,200` 个 description-code pair 进行测量。结果显示 `9.93%` 的 pair 存在不一致，且问题不仅是功能误报，还包括未声明副作用。

**影响与处置**  
这说明仅靠 MCP 工具自然语言描述做选择或授权是有盲区的。代码 Agent 安全基线至少要补三项：  
1. 工具描述与实现的一致性扫描；  
2. 对文件写入、网络访问、命令执行等高风险副作用做声明校验；  
3. 把“描述变更但代码未审计”作为审批或隔离触发条件。

**信息来源**  
- https://arxiv.org/abs/2606.04769
- https://export.arxiv.org/list/cs.CR/recent

## Stored Prompt Injection 从单会话升级到跨会话持久化，攻击面覆盖记忆、文件系统与工具状态

**入选理由**  
论文直接命中 `tool attack`、`prompt injection` 和长期记忆污染，是当前 Agent 系统最现实的系统级风险之一。

**技术细节**  
`What If Prompt Injection Never Left? Exploring Cross-Session Stored Prompt Injection in Agentic Systems` 借鉴 stored XSS 的概念，提出“跨会话存储型 prompt injection”。核心观点是：一旦恶意内容被写入 Agent 的长期状态，例如 memory、filesystem、tool artifact 或其他持久上下文，它就能在原始攻击交互结束后继续影响后续执行。作者给出了 persistence channel taxonomy，并构建 benchmark 与 sandbox toolkit，用于量化不同模型、不同攻击目标和不同持久化通道下的攻击成功情况。

**影响与处置**  
把 prompt injection 只当作“本轮对话污染”已经不够。工程上应把 memory write、文件落地、工具输出入库视为高风险写路径，并补上内容分级、来源标记、TTL、人工确认和回滚机制；否则撤回恶意输入后，污染仍可能留在长期状态里。

**信息来源**  
- https://arxiv.org/abs/2606.04425
- https://export.arxiv.org/list/cs.CR/recent

## Memory Poisoning 攻击被系统化：4 条写入通道、9 类结构脆弱点、6 类攻击

**入选理由**  
相比泛化“记忆有风险”的讨论，这篇论文给出了结构化攻击面枚举和专用基准，适合作为 Agent 长期记忆防护的参考框架。

**技术细节**  
`From Untrusted Input to Trusted Memory: A Systematic Study of Memory Poisoning Attacks in LLM Agents` 把 persistent memory 视为独立攻击面。作者识别了 `4` 条 memory write channel、`9` 类来自模型能力/系统提示/架构设计的结构性脆弱点，以及 `6` 类 memory poisoning 攻击，并提出基准 `MPBench`。论文指出：写入和检索 memory 越积极的 Agent 越容易被利用，而现有 prompt injection 防御对 memory poisoning 覆盖不足。

**影响与处置**  
这意味着“让 Agent 多记忆”本身就是安全权衡。默认策略应从“尽量多写”改为“最小必要写入”，并为记忆项增加来源、可信度、创建工具、审批状态、失效时间和删除链路等审计字段。

**信息来源**  
- https://arxiv.org/abs/2606.04329
- https://export.arxiv.org/list/cs.CR/recent

## Agent Tracing/Provenance 进入系统化梳理阶段，审计重点从最终答案转向过程级证据

**入选理由**  
命中固定关键词 `agent tracing`。虽然这是一篇综述，但它把 evidence tracing、tool-use provenance、memory lineage 和 runtime guardrail 放到同一框架下，适合直接指导审计字段设计。

**技术细节**  
`From Agent Traces to Trust: Evidence Tracing and Execution Provenance in LLM Agents` 讨论的不只是日志，而是把检索证据、工具输出、记忆项、环境观察、中间结论、动作与最终答案之间的关系建模。论文给出 provenance taxonomy，覆盖 trace source、evidence unit、relation、granularity、timing、representation 和 trust function，并专门讨论 tool-use provenance、provenance-bearing memory、trace-based observability 与 failure diagnosis。

**影响与处置**  
对代码 Agent、CLI Agent、IDE Agent 与 MCP server 来说，最低限度的日志字段应包括：工具入口、参数摘要、资源访问范围、命令执行结果、凭据使用点、外部内容到工具参数的传播路径，以及人工审批/回滚记录。否则即便模型最终输出正确，也难以审计“为什么这样做”与“是否越权”。

**信息来源**  
- https://arxiv.org/abs/2606.04990
- https://export.arxiv.org/list/cs.CR/recent
