统计窗口以 2026-06-07 运行时点回看过去约 24-36 小时。经对 OpenAI、Anthropic、Google、xAI、Meta、Mistral、DeepSeek、Qwen 等官方索引做轻量核验，未发现满足“旗舰/准旗舰模型、关键 API 能力、定价、上下文长度、Agent/工具调用能力”门槛且落在本窗口内的头部厂商官方实质更新；正文仅保留可核实的技术论文热点。

## ToolChoiceConfusion：把工具暴露从 100 个压到每步 1 个，直接针对 Agent 误调用

**入选理由**  
2026-06-05 挂到 arXiv cs.AI recent，主题直接命中 tool use/agent reliability，且给出了可复现实验规模，不是泛化“提示工程优化”。

**技术细节**  
论文提出 `Causal Minimal Tool Filtering (CMTF)`，不是按语义相似度筛工具，而是依据“当前状态到目标状态所需的最小因果前沿”暴露下一步工具。作者在 102 个多步任务、100 个工具、4 个 LLM 后端、2448 组 task-method-model 运行上对比 all-tools exposure、关键词检索和 state-aware filtering。结果是：在总体成功率与最强因果基线持平的同时，把每步可见工具从 100 个压到 1 个，且相对全量工具暴露把 token 使用量降约 90%。

**影响与处置**  
这类方法对代码 Agent、企业工作流 Agent 和 MCP 工具集都很实际，因为它不要求重训模型，只要求给工具补充前置条件/效果约束。短期可观察点是：现有 Agent 框架是否开始把工具元数据从“自然语言描述”升级为可机读的 precondition/effect contract。

**信息来源**  
- https://arxiv.org/abs/2606.06284
- https://export.arxiv.org/list/cs.AI/recent

## Vortex：把稀疏注意力从论文概念推进到可编程服务栈，面向长上下文 Agent 吞吐优化

**入选理由**  
2026-06-05 出现在 arXiv cs.AI recent。相较常见“新注意力机制”论文，这篇更偏系统落地，直接报告在真实服务栈上的吞吐增益。

**技术细节**  
Vortex 提供一个嵌入 Python 的前端语言和 page-centric tensor 抽象，用来表达不同稀疏注意力算法，并把后端直接接到现代 LLM serving stack。作者声称 AI agent 可以用 Vortex 自动生成并迭代稀疏注意力算法；最优方案在保持精度的前提下，相对 full attention 达到最高 `3.46x` 吞吐提升。论文还把方法扩展到更难实验的架构和超大模型，在 NVIDIA B200 上对 MLA 架构 `GLM-4.7-Flash` 报告最高 `4.7x` 吞吐提升，对 `229B` 参数 `MiniMax-M2.7` 报告 `1.37x` 提升。

**影响与处置**  
如果这些结果能在主流 serving 框架复现，受益最大的不是通用聊天，而是长上下文、工具密集、需要多轮状态保留的 Agent 任务。落地时应重点核实两点：一是准确率保持条件是否依赖特定工作负载；二是稀疏策略是否会对检索、代码补全和多工具规划引入系统性偏差。

**信息来源**  
- https://arxiv.org/abs/2606.06453
- https://export.arxiv.org/list/cs.AI/recent
