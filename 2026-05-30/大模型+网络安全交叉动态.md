## AgentDoG 1.5：面向 OpenClaw/Codex 执行场景的轻量级 Agent 安全对齐框架

**热度判断：**该论文 2026-05-28 提交至 arXiv，覆盖 cs.AI、cs.CL、cs.CR、cs.LG 等分类，并在 Hugging Face Daily Papers 2026-05-29 榜首出现，抓取时页面显示 81 个社区点赞，属于“Agent 执行安全”专题候选。其关键词直接命中 OpenClaw、agent safety、tool guardrail、real-time moderation 和交互式执行环境。

**技术细节：**论文提出 AgentDoG 1.5，用更新后的 Agent 安全风险分类覆盖 Codex 与 OpenClaw 执行场景；通过 taxonomy-guided data engine 和 influence-function purification，用约 1k 样本训练 0.8B、2B、4B、8B 轻量模型；同时构建 agentic safety SFT/RL 训练环境，声称在 Docker 级环境中将部署开销降低两个数量级。AgentDoG 1.5 还可作为无需训练的在线 guardrail，对实时 Agent 执行进行安全审核。

**影响与处置：**该方向适合落在“高权限 Agent 的旁路安全层”：在 MCP server、浏览器/GUI Agent、代码执行器、自动化渗透测试框架等场景中，对工具参数、上下文来源、外部网页内容、长期记忆和执行结果进行实时审核。落地时应重点验证误杀率、延迟、对自适应攻击的鲁棒性、对中文/多语言提示注入的覆盖，以及 guardrail 被工具输出污染后的失效模式。

## Claude Opus 4.8 系统卡暴露 Agentic Safety 重点：prompt injection 仍是长任务 Agent 的关键风险

**热度判断：**Anthropic 系统卡把 Agentic safety、恶意 Claude Code 使用、恶意 computer use、prompt injection across surfaces、browser use、coding 等列为独立章节。由于 Opus 4.8 与 dynamic workflows 同日发布，安全评估应与产品能力一起解读。

**技术细节：**系统卡称 Opus 4.8 在未加 safeguards 时多项 cyber eval 强于 Opus 4.7，但在若干 Agent 上下文中，包括 prompt injection 攻击鲁棒性，弱于 Opus 4.7；Anthropic 表示 safeguards 能在实践中缩小差距。Dynamic workflows 允许数十到数百个 subagents 在单次任务中并行执行，适合代码库级迁移、安全审计和优化审计，但同时扩大了外部内容注入、工具参数污染、长期任务状态漂移和权限继承的风险面。

**影响与处置：**对企业采用 Claude Code、Claude API 或云平台托管 Opus 4.8 的团队，建议把 Agent 执行链路拆成“输入来源、规划、工具选择、参数、执行结果、合并、最终输出”七类审计对象；对 shell、网络、文件写入、凭据访问、PR 创建和部署类工具设置审批；对网页、issue、README、依赖包脚本和测试输出标注不可信来源；在组织层面限制 auto mode 与 dynamic workflows 的使用范围。

## MCP 与 Agents SDK 安全基线：授权、同意、工具审批和 tracing 应成为默认控制

**热度判断：**过去 24 小时未发现来自 MCP 规范或 OpenAI Agents SDK 的重大新版本公告，但固定关键词检索继续出现 MCP server prompt injection、tool approval、agent tracing、tool guardrail 等讨论。结合 Opus 4.8 动态工作流和 AgentDoG 1.5，MCP/Agent 工具边界仍是本日报的交叉安全观察点。

**技术细节：**MCP 官方规范将工具视为可能执行代码的能力，强调用户同意、数据访问控制、显式工具授权和 LLM sampling 请求审批；授权教程使用 OAuth authorization code、PKCE、scope、state、access token/refresh token 等机制，并要求代理服务器按 client 维度处理 consent，避免 confused deputy。OpenAI Agents SDK 文档提供 input/output guardrails 和 tracing：guardrail 可在工具入参或工具输出中阻断敏感模式，tracing 记录 LLM generations、tool calls、handoffs、guardrails 和自定义事件。

**影响与处置：**MCP server 不应把“接入成功”视为“可无限代理所有权限”。建议为每个 client、server、tool 和 scope 建立最小权限授权；把工具调用审批与参数 diff 展示给用户；对外部内容进入上下文的位置加不可信标签；对 tool output 做敏感信息扫描；在 traces 中保留 request id、server id、tool id、scope、approval action、arguments hash、结果摘要和错误码，便于事后追溯。

## Metasploit 新增 Ollama 扫描模块，LLM 本地服务进入攻防工具链枚举面

**热度判断：**Rapid7 2026-05-29 Metasploit 周报新增 `scanner/http/ollama_info` 辅助模块，用于枚举已安装 LLM 及其细节。虽然不是漏洞披露，但它说明本地 LLM 服务正在被纳入常规资产发现和攻防工具链。

**技术细节：**该模块面向 Ollama HTTP 服务，定位为 auxiliary scanner，用于识别环境中安装的模型和相关信息。周报同批还加入 Citrix NetScaler CVE-2026-3055 scanner、DirtyFrag Linux LPE 模块、dompdf RCE 和 WordPress Supsystic Contact Form SSTI RCE 模块。

**影响与处置：**企业内部 Ollama、vLLM、LM Studio、Open WebUI 等本地模型服务通常承载源码、文档和内部数据。建议将其纳入 ASM/CMDB，限制监听地址，强制认证和网络分段，记录模型拉取来源与插件/工具权限；对暴露在办公网或公网的推理端口执行周期性扫描。

**信息来源：**

- Rapid7：Metasploit Wrap Up 05/29/2026，https://www.rapid7.com/blog/post/pt-metasploit-wrap-up-05-29-2026/
